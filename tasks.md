Wired into the `catch` block of `handleGenerateCoachNote`, so a technical failure (network error, `/api/ai-coach` returning `not_configured`, or `upsertCoachNote` failing) always refunds the 50 coins before surfacing the error — coins are only permanently spent on a successful, cached analysis.

## Strict Daily Habit Locking

- [x] **Generalized the lock window:** `db.ts` previously had `isHabitLoggable(habit, dateStr, now)`, which only enforced a "log date 00:00 → next-day 09:00" window for habits *with* `scheduled_start`, and let any *unscheduled* habit be logged for any past date reachable via the 30-day week-navigation. Replaced it with `isLogDateLocked(dateStr, now)` — the same window math, but applied universally to every habit log regardless of type/schedule, per the spec's "Lock Deadline = current date's next day at 09:00" rule. Net effect: today is always editable; yesterday is editable only before 09:00 today; anything older is always locked — matching "cannot modify logs prior to the current day unless it's before 09:00 AM the next day" exactly. **Scope note:** this materially narrows the previously-documented "retroactive marking for previous days" behavior down to effectively "today + a same-morning grace window for yesterday" — `PROJECT.md` updated to reflect this.
- [x] **Server-side enforcement (not just UI):** `toggleHabitLog()` now throws `LOG_LOCKED_MESSAGE` if the target `log_date` is locked, checked *before* the upsert. New `deleteHabitLog(habitId, userId, date)` centralizes the "reset to pending" delete path (same lock check) — `HabitsLog.tsx` no longer calls `supabase.from('habit_logs').delete()` directly anywhere (3 call sites replaced), so every mutation path (including a stale offline-queue replay via `flushPendingLogs`) is now checked at the data layer, not just gated by a client-side `disabled` attribute.
- [x] **Frontend — toggling completion:** `HabitsLog.tsx` computes `dateLocked = isLogDateLocked(selectedDate)` once per render. `toggleHabit`, `markHabitMissed`, and `markNegativeHabit` all early-return with a `sonner` toast (`log_locked_error`) if `dateLocked`. Checkboxes/X-buttons across all three sections (scheduled timeline, unscheduled list, negative habits) are `disabled` and dimmed when locked; the small "mark as broken/missed" X buttons are replaced with a static 🔒 `Lock` icon instead of a dead button.
- [x] **Frontend — editing habit (log) details:** `handleValueChange` (the +/− steppers for numeric habits, e.g. "3/5 stakan") gets the same `dateLocked` guard and, when locked, the steppers are replaced with a "🔒 Qulflangan" label instead of interactive controls — satisfies "applies to both toggling completion and editing habit details." **Scope decision:** interpreted "habit details" as the day's *log* value, not the habit's global definition (name/emoji/target in `HabitsManager.tsx`) — renaming/retargeting a habit isn't date-scoped and isn't part of the Daily Journal locking concept the spec is about.
- [x] **Locked-day indicator:** new banner ("🔒 Bu kun qulflangan...") shown above the progress bar whenever `dateLocked` is true, so a user browsing an old date via the week navigator immediately understands why nothing responds to taps, in addition to the per-row toast/disabled state.
- [x] **Verification:** `npx tsc --noEmit` and `npm run build` both clean. Confirmed zero remaining references to the old `isHabitLoggable` name or direct `supabase.from('habit_logs').delete()` calls in `HabitsLog.tsx`.

## Numeric Habit Metrics — Save Logic

- [x] **Found the real bug:** the "Raqamli Ko'rsatkichlar" sliders (`NumericMetrics`/`MetricSlider` in `HabitsLog.tsx`) were never wired to persistence at all — no `profile`/date props, no DB read or write, pure local `useState(0)` that reset on every remount. Not "saves on every change, want a Save button instead" — there was no save path whatsoever.
- [x] **Storage decision (deviates from the spec's literal signature):** the spec suggested `upsertHabitMetric(userId, habitId, logDate, value)`, but "Tabiat vaqti" / "Ijtimoiy tarmoq vaqti" aren't tied to any real habit (no `habit_id` — `NumericMetrics` doesn't even receive the habit list). Per this app's existing "one table per kind of daily metric" convention (`health_logs` already unifies `sleep_hours`/`screen_time_hours` as the single source of truth for Health + Journal), added `nature_time_minutes`/`social_time_minutes` columns to `health_logs` instead (migration `019_numeric_metrics.sql`) and extended `upsertHealthLog`'s partial-update `fields` type to include them.
- [x] **`upsertHabitMetric(userId, metricKey, logDate, value)` in `db.ts`:** kept the requested function name/shape (per-metric write, called from the UI) but swapped `habitId` for a `HabitMetricKey` (`'nature_time_minutes' | 'social_time_minutes'`) since that's what actually identifies which slider this is; internally delegates to `upsertHealthLog`. Also enforces the same `isLogDateLocked` check as the rest of the Daily Journal (added last round) — metrics for a locked day can't be saved either, consistent with the rest of the strict-locking feature.
- [x] **State management:** `NumericMetrics` now takes `profile`/`logDate`/`locked` props (threaded from `HabitsLog.tsx`: `logDate={selectedDate}`, `locked={dateLocked}`), loads the existing saved values via `getHealthLog` on mount/date-change, and tracks `value` (slider position) separately from `saved` (last-persisted value) per metric — the delta between them is the "dirty" flag.
- [x] **Save button + feedback:** each `MetricSlider` gets its own Save button (not a single "save all" — the two metrics are independent, per the spec's "enabled only when there are unsaved changes for **that** metric"). Disabled unless `dirty && !saving && !disabled`; shows a spinner while saving, `toast.success('log_metrics_saved')` on success, `toast.error(...)` with the thrown message (or a generic fallback) on failure — same `sonner` pattern used elsewhere in this file. The +/− steppers, range input, and Save button are all disabled (and the whole card dimmed + a 🔒 badge shown) when `locked` is true.
- [x] **Verification:** `npx tsc --noEmit` and `npm run build` both clean.
